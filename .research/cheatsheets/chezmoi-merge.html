How the 3-way merge works in VS Code, and why some differences are invisible.

1The 3-Way Merge Model

A 3-way merge compares two versions against a common ancestor (the base). It does not diff left vs. right — it diffs left vs. base and right vs. base independently.

base
last committed version in git
chezmoi
~/.local/share/chezmoi/
home
~/

The base is the last git commit of the chezmoi file. Each side is compared against the base, not against each other.

💡
The base comes from git show HEAD:<file>. If the file was never committed, the merge degrades to a 2-way diff (less useful). Always commit before editing.

2VS Code Merge Layout

The merge editor opens with three visible areas. The base is invisible — VS Code uses it internally.

config.toml — 3-way Merge Editor
Current chezmoi — what chezmoi wants
Left pane
Incoming home — what's at ~/
Right pane
Result — the chezmoi file that gets saved
Starts as a copy of the left pane (chezmoi)
⚠️
The result starts as a copy of the LEFT (chezmoi) side. This is the key to everything below. Chezmoi's changes are "free" — already in the result. Home's changes are "incoming" and need explicit acceptance.

3Scenarios

Each side changed a different line

Chezmoi changed theme. Home changed plugins. Base has the originals of both.

theme — the invisible change
Left says "modified", right says "gruvbox".
They differ — but there is no highlighting, no Accept buttons. Completely invisible.
Why? Chezmoi changed it, home didn't. The result starts from the left, so "modified" is already there. Nothing to do.
plugins — the highlighted change
Both panes show highlighting and Accept buttons.
The result still has "docker". You need to click Accept Incoming on the right to get "k8s".
Why? Home changed it, chezmoi didn't. Home's changes are "incoming" and need a click.
🔍
The deceptive part: theme clearly differs between the two panes, but there is zero visual indication. You can only see it if you read the actual values line by line. The merge UI is correct — it's showing you what changed relative to the base, not what differs between panes.

⚠️
This is a real conflict. Both sides changed theme relative to the base. Both panes highlight it with Accept buttons. You pick one, accept a combination, or edit the result manually.
↔️
Compare with Scenario 1. In both scenarios, highlighted lines have Accept buttons on both panes. The difference is why: Scenario 1's plugins line shows buttons because home's change is "incoming." Scenario 2's theme line shows buttons because both sides genuinely changed. Visually they look the same.

4The Rule

The result starts from the left (chezmoi). This creates an asymmetry:

Chezmoi changed? Home changed? What VS Code does Visual
Yes No Already in result. No highlighting. Invisible. invisible
No Yes Accept buttons on both panes. Click Accept on right to bring it in. accept needed
Yes Yes Accept buttons on both panes. Real conflict — you choose. conflict
No No Nothing to do. Identical everywhere. nothing
🔑
"Changed" means "differs from the base" (last committed version) — not "differs from the other pane." This is why row 1 and row 2 look completely different in the UI even though both are "one side changed."
✔️
Chezmoi changes
Free. Already in the result. No action needed. Can be easy to miss when reviewing.
👉
Home changes
Incoming. Need a click to accept. Always visible with Accept buttons and highlighting.

5After the Merge

chezmoi merge only modifies the chezmoi file. Home (~/) is untouched. You must apply afterwards.

chezmoi merge ~/file
Resolve conflicts
Saves to chezmoi
→
chezmoi apply ~/file
Push merged result
to home
→
chezmoi status ~/file
Should be clean
‼️
Don't forget chezmoi apply. After merge, chezmoi has the resolved version but home still has the old file. Without apply, your merge work just sits in the repo.
Quick decision table
I want to... Command
Keep chezmoi, discard home changes chezmoi apply ~/file
Keep home, discard chezmoi changes chezmoi re-add ~/file
Keep parts of both chezmoi merge ~/file then chezmoi apply ~/file
See what's different first chezmoi diff ~/file
